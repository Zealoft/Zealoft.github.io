在CentOS 7 下静态链接MySQL库

发表于 2018-10-12 | 更新于 2018-10-13 | 分类于 Linux

写在前面的一些废话

什么是库？能吃吗？

事实上，在windows平台和linux平台下都大量存在着库。
本质上来说库是一种可执行代码的二进制形式，可以被操作系统载入内存执行。
由于windows和linux的本质不同，因此二者库的二进制是不兼容的。
本文仅限于介绍linux下的库。

库的意义是什么？

库是别人写好的现有的，成熟的，可以复用的代码，你可以使用但要记得遵守许可协议。
现实中每个程序都要依赖很多基础的底层库，不可能每个人的代码都从零开始，因此库的存在意义非同寻常。
共享库的好处是，不同的应用程序如果调用相同的库，那么在内存里只需要有一份该共享库的实例。

库文件有什么规范吗？

在linux下，库文件一般放在/usr/lib /lib下。而在我的CentOS 7环境里，库文件大多数被放在了/usr/lib64目录下。
静态库的名字一般为libxxxx.a，其中xxxx是该lib的名称
动态库的名字一般为libxxxx.so.major.minor，xxxx是该lib的名称，major是主版本号， minor是副版本号

什么是静态链接？什么是静态库？和动态库的区别是什么？

要回答这个问题，我们首先来看一看什么是静态库。
在Linux下有两种库，一种是静态库（.a），一种是动态库（.so）（也被叫做共享库），其对应着不同的生成与使用方式。二者的不同点在于代码被载入的时刻不同。
静态库的代码在编译过程中已经被载入可执行程序，因此体积较大。
共享库的代码是在可执行程序运行时才载入内存的，在编译过程中仅简单的引用，因此代码体积较小。

怎么让系统找到我写的库？

如果安装在/lib或者/usr/lib下，那么ld默认能够找到，无需其他操作。
如果安装在其他目录，需要将其添加到/etc/ld.so.cache文件中，步骤如下
1.编辑/etc/ld.so.conf文件，加入库文件所在目录的路径
2.运行ldconfig，该命令会重建/etc/ld.so.cache文件

如何知道一个可执行程序依赖哪些库？

ldd命令可以查看一个可执行程序依赖的共享库。
例如，现在有一个test.c文件，代码十分简单

//test.c
#include <stdio.h>
int main()
{
    printf("Hello,world\n");
    return 0;
}

gcc/g++静态编译环境配置

gcc静态编译环境配置

yum安装glibc-static库
yum install glibc-static
如果不安装上述库，事实上也可以用gcc静态编译简单的c文件，实现方法是使用locate命令在系统中搜索libc.a文件
locate libc.a
将该路径拷贝并在gcc中加上-L参数，就可以编译成功
假设路径是

/usr/lib/x86_64-redhat-linux6E/lib64 -static

那么可以使用gcc命令

$ gcc main_printf.c -o main_printf_static -L/usr/lib/x86_64-redhat-linux6E/lib64 -static

生成可执行文件后，使用ldd命令查看

ldd main_printf_static

会发现其提示不是动态可执行文件，说明该可执行文件所有的库都已经静态链接到其中。
使用size命令查看可执行文件大小

size main_printf_static

会发现该可执行文件大小远远大于正常生成的可执行文件。

但使用这种方法，使用g++编译c++文件时就会产生一系列错误

在使用上述yum命令安装glibc-static库后， 静态编译C文件就不再需要额外的-L选项。

g++静态编译环境配置

运行yum安装libstdc++库

yum install libstdc++-static

安装完后再次编译，就可以顺利通过编译。

至此，CentOS 7环境的gcc/g++静态编译环境配置顺利完成。

使用g++静态链接MySQL库到可执行文件中

使用g++编译含有MySQL函数的代码

安装mysql c++库
todo

假如当前目录有cpp文件mysql_demo.cpp，其中调用了MySQL的API。
首先使用g++编译

g++ -c mysql_demo.cpp \
    $(shell mysql_config --cflags)

值得一提的是，MySQL提供了路径的shell config命令，上述参数$(shell mysql_config --cflags)就是调用了该条命令，把所需的库自动加入到g++命令当中。
然后使用g++生成可执行文件

g++ -o mysql_demo mysql_demo.o \
    $(shell mysql_config --libs)

此时的可执行文件是所有的库都是动态链接，我们的目的就是将其改为全静态链接

使用g++静态链接含有MySQL函数的代码

首先尝试直接在g++编译命令后面加上-􏰂􏰇stat􏰇ic，发现报大量错误，此处不一一截图。

尝试安装缺失的库

yum install openssl-static
yum install zlib-static

安装完后再次编译，发现错误有一定程度的减少，但无法完全消除。一个难以解决的问题是ld程序不断提示缺少krb5库，查询后发现该库已经不再提供static库。
这种情况下，我做出了两种选择：

Choice 1 放弃全静态编译，只静态链接MySQL的部分

具体的方法是在g++命令中显式指定/usr/lib64中的libmysqlclient.a而不加-static选项进行编译。部分选项如下

g++ mysql_demo.o -o mysql_demo /usr/lib64/libmysqlclient.a

此时可以生成可执行文件，且对该可执行文件使用ldd命令，发现其中已经不含与mysql有关的库。

Choice 2 放弃该版本的MySQL，在线下载mysqlclient.a文件

朋友告诉我，想要全静态链接MySQL库，需要使用特定版本的libmysqlclient.a文件。因此在下列网址下载mysql-connector-c-devel-6.1.11-1.el6.x86_64.rpm文件
http://ftp.ntu.edu.tw/MySQL/Downloads/Connector-C/
解压得到mysql-connector-c-devel.cpio文件，查询资料后发现其依然是压缩文件，使用cpis -idmv < mysql-connector-c-devel.cpio命令解压后得到usr文件夹

其中的lib64/mysql文件夹中含有libmysqlclient.a文件，将其发送到Linux目录下，设定链接路径后发现可以顺利使用-static选项进行编译。

g++ -c mysql_demo.cpp $(mysql_config --cflags)
g++ -o mysql_demo mysql_demo.o ./libmysqlclient.a /usr/lib/gcc/x86_64-redhat-linux/4.8.2/libstdc++.a -lpthread -lssl -lcrypto -ldl $(LIB_Z) -static

编译后用ldd命令查看可执行文件，提示不是动态可执行文件。
